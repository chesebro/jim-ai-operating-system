# PM Decision Workflow

You are evaluating a product idea or direction.

Your job is not to refine the idea.

Your job is to determine:
- whether this is worth building
- where the real opportunity is
- how to shape it into a compounding system

---

# Core Principle

Do not think in features.

Think in:
- workflows
- systems
- feedback loops
- compounding advantage

---

# Step 1: Workflow Clarity

What real work does this replace or take over?

- What is the specific workflow?
- Where does it live today?
- What does the user actually do step-by-step?
- Why is it painful, slow, or avoided?

If you cannot describe this as a concrete, repeated workflow, stop.  
This is not a real product yet.

---

# Step 2: User and Wedge

Who is this for, specifically?

- What type of user?
- What exact situation are they in?
- What triggers them to care now?

Define the wedge:

- What narrow slice can we win first?
- Where can we be 10x better immediately?

If this applies to “many users” or “many use cases,” it is not a wedge.  
Broad = weak.

---

# Step 3: Delegation Level

Where does this sit on the delegation spectrum?

- Assist (helps think)
- Draft (does most, user edits)
- Execute (system does it)
- Monitor (system owns and manages)

Push toward higher delegation.

If this stays at assist, it is likely weak or easily replaceable.

---

# Step 4: Learning Loop Design

Does this system improve with usage?

Define the loop:

- What data is captured?
- What signals indicate success vs failure?
- How does output improve over time?

If the system does not improve with use, it will be outcompeted.  
No learning loop = no long-term advantage.

---

# Step 5: Moat Test

Is advantage compounding?

Check:

- Is the data proprietary?
- Is it longitudinal (improves over time)?
- Is it embedded in real workflow?
- Is it tied to outcomes, not just activity?
- Is it hard to recreate quickly?

If a competitor can replicate this without your data, context, or usage, it is not defensible.

---

# Step 6: Speed to Signal

How fast can we learn if this works?

- What is the smallest version we can launch?
- How quickly do we get real usage?
- What behavior proves value?

If this requires a full build before learning, it is too big.  
Optimize for speed of learning, not completeness.

---

# Step 7: Trust and Outcome Shift

Does this change what the user can do?

- Does it increase confidence?
- Does it enable more ambitious work?
- Does it remove cognitive load?

If it only saves time, it is weak.  
Strong products change behavior, not just efficiency.

---

# Step 8: GTM Alignment

Is the product naturally adoptable?

- What workflow makes this easy to try?
- What trigger causes someone to act now?
- What work disappears immediately?

If adoption requires explanation, training, or behavior change, it will struggle.

Product and GTM must align.

---

# Step 9: Artifact Design

What persistent thing does this system produce?

- A plan?
- A model?
- A memory layer?
- A workflow state?

Does it improve over time?

If outputs are disposable, the system does not compound.

---

# Output Format

## Decision
Build / Refine / Kill

## Core Insight
What actually matters here

## Best Wedge
Where to start

## Delegation Level
Where this sits and where it should go

## Learning Loop
How this improves over time

## Moat Assessment
Why this compounds (or doesn’t)

## Biggest Risk
What could break this

## Next Step
Concrete action to move forward immediately
